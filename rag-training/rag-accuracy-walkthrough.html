1 · Is your data already structured before query time?
Yes (you) → retrieval is half-solved. Lean on metadata filters + hybrid search over typed fields.
No → fix ingestion first. No reranker saves a bad chunking strategy.
2 · Do answers need multi-hop relational reasoning?
Yes → add Graph RAG for those query types. Relationships beat similarity here.
No → skip the graph. It's complexity you don't need.
3 · Is accuracy worth 5–10× the latency & cost?
Yes (policy/compliance) → wrap an agentic verify-and-cite loop around retrieval.
No (chatty UX) → Hybrid + Rerank alone, single pass.

Hybrid retrieval + metadata filtering
BM25 (exact terms) + vector (meaning), fused. Filter by schema fields before ranking.
3
Cross-encoder reranking
Retrieve top-50 → rerank to top-5. The single highest-ROI accuracy lever.
4
Agentic verify-and-cite loop
Grade the context, re-query if weak, self-check the answer against the question, attach citations.
5
Faithfulness evaluation in CI
RAGAS faithfulness > 0.9 as a gate. If you can't measure it, you can't claim it.
The recommendation
For an accuracy-first system on already-structured schema data, make Hybrid + Rerank with metadata filtering your backbone, and wrap an agentic verify-and-cite loop around it. Reach for Graph RAG only when queries are genuinely multi-hop relational.
Naive RAG is never the answer when accuracy is the priority. And "Agentic RAG" on its own isn't a pattern you pick instead of the others — it's a control loop you wrap around a strong retrieval stack.

All of this runs offline — before any user types a question. It's a kitchen prep line: you do the chopping once so that at dinner service (query time) every order is fast.
text
the chunk's words, as readable text. This is what keyword search reads.
embedding
the same chunk as ~1,536 numbers (a vector). Stored with pgvector. This is what meaning search reads.
metadata
labels: region, policy type, date, doc id. Stored as JSONB. This is what filtering reads.
-- one row = one chunk, three useful shapes of the same thing
CREATE TABLE chunks (
  id        bigserial,
  text      text,            -- words  → keyword search
  embedding vector(1536),    -- numbers → meaning search (pgvector)
  metadata  jsonb            -- labels  → filtering
);

🔑 Keyword door

reads the text column
  • Matches the actual words you typed
  • Built with a tsvector index (Postgres full-text)
  • Unbeatable for exact things: codes, names, defined terms

🧭 Meaning door

reads the embedding column
  • Matches by meaning, even with no shared words
  • Built with a pgvector index (HNSW or IVFFlat)
  • Catches paraphrases: "get my money back" → "refund policy"
"Dual indexing" just means two separate indexes over the same rows — one per door. You run both searches and merge the results. That merged search is what people call hybrid.

Keyword search needs to rank matches. TF/IDF is the classic rule for "how much should this word count?" Watch the weight change as a word gets rarer:

TF = Term Frequency: how often the word appears in this document (more = more relevant). IDF = Inverse Document Frequency: how rare the word is across all documents (rarer = more meaningful). Multiply them. So a rare word that appears a lot in one doc scores highest; a word like "the" scores ~nothing.

BM25 is the modern, smarter cousin of TF/IDF — same idea, but with two fixes: it stops repeated words running away with the score ("saturation"), and it adjusts for document length. ParadeDB (which ships the pg_search extension) is a Postgres add-on that gives you real BM25 inside your database. Plain Postgres only gives you the "good enough" TF/IDF version.
HNSW = Hierarchical Navigable Small World. Every vector is a dot connected to a few nearby dots, forming a web. To find the closest match, search starts at an entry dot and "walks" along the links, always stepping toward the query — reaching the answer in a few hops instead of comparing against millions of dots. Fast, very accurate (high "recall"), uses more memory. It's the default choice today.
IVFFlat takes the other approach: it first sorts all vectors into a few clusters ("cells"). At query time it figures out which cell the query falls in and only searches that cell — ignoring the rest. Faster to build and lighter on memory than HNSW, but it can miss a true match sitting just over a cell border (slightly lower recall). Rule of thumb: HNSW for quality, IVFFlat for tight memory budgets.
